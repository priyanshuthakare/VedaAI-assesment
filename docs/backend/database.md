# Database Schemas

The backend uses **Mongoose** with **MongoDB**. Data is stored across two collections: `assignments` and `results`.

---

## Collection: `assignments`

**File:** `backend/src/models/Assignment.ts`

Stores the teacher's input configuration for every assignment request.

### Schema

```typescript
{
  input: {
    topic:            String,   // Required. e.g. "Photosynthesis"
    totalQuestions:   Number,   // Required. Total questions across all types
    marksPerQuestion: Number,   // Required. Default marks per question
    questionTypes: [{
      type:     String,  // Required. "mcq" | "short" | "long" | "fill_blank" | "true_false"
      count:    Number,  // Required. How many questions of this type
      marksEach:Number,  // Required. Marks allocated per question of this type
    }],
    difficulty: {
      easy:   Number,  // Required. Percentage, e.g. 40
      medium: Number,  // Required. Percentage, e.g. 40
      hard:   Number,  // Required. Percentage, e.g. 20
    },
    dueDate:      String,   // Optional. "DD-MM-YYYY" format
    instructions: String,   // Optional. Additional context for the AI
    fileContent:  String,   // Optional. Extracted text from uploaded files
  },
  status:   "queued" | "processing" | "completed" | "failed",  // Default: "queued"
  resultId: String,   // Set after generation completes. References results._id
  error:    String,   // Set if generation fails. Contains the error message.
  createdAt: Date,    // Auto-generated by Mongoose timestamps
  updatedAt: Date,    // Auto-generated by Mongoose timestamps
}
```

### Notes
- `status` is updated by the BullMQ worker at each stage of processing.
- `resultId` links to the `results` collection — a soft reference (string, not ObjectId) for flexibility.
- The `difficulty` object uses percentage values that must sum to 100 to produce a balanced paper.

## Collection: `results`

**File:** `backend/src/models/Result.ts`

Stores the fully structured question paper generated by the AI. This is the data rendered on the Output page and used to generate the PDF.

### Schema

```typescript
{
  assignmentId: String,   // Required. Indexed. References the Assignment._id

  questionPaper: {
    title:      String,  // Required. Paper title, e.g. "Photosynthesis Assessment"
    subject:    String,  // Required. Subject name, e.g. "Biology"
    totalMarks: Number,  // Required. Sum of all question marks
    duration:   String,  // Required. Suggested exam duration, e.g. "60 minutes"

    sections: [{
      id:          String,  // Required. Section identifier, e.g. "A"
      label:       String,  // Required. Display name, e.g. "Section A"
      instruction: String,  // Required. e.g. "Attempt all questions. Each carries 2 marks."

      questions: [{
        id:         String,                           // Required. Unique question ID
        number:     Number,                           // Required. Sequential question number
        text:       String,                           // Required. The question text
        difficulty: "easy" | "medium" | "hard",      // Required
        marks:      Number,                           // Required
        type:       "mcq" | "short" | "long" | "truefalse",  // Required
        options:    [String],                         // Only present for MCQ questions
      }]
    }]
  },

  createdAt: Date,   // Auto-generated
  updatedAt: Date,   // Auto-generated
}
```

### Notes
- `assignmentId` has a MongoDB index for fast lookups by the API and worker.
- `options` is only populated for MCQ questions; other question types omit this field.
- An `answer` field is generated at query-time inside the PDF controller from the AI output — it is not stored in the database to keep the schema clean.
- The `sections` array directly maps to the rendered output on the frontend.
